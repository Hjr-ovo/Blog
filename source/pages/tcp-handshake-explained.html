为什么需要握手和挥手？

理解 TCP 可靠传输的基础
TCP 是什么？

TCP（Transmission Control Protocol，传输控制协议）是互联网中最核心的协议之一。它是一个面向连接的、可靠的传输层协议。 在发送数据之前，通信双方必须先建立连接；数据传输结束后，需要断开连接。这就是三次握手和四次挥手的作用。

核心概念：序列号（Sequence Number）

TCP 之所以"可靠"，关键在于每发送一个字节的数据，都会分配一个序列号（seq）。 接收方通过确认号（ack）告诉发送方："我已经收到了 seq=N 之前的所有数据，请从 seq=N+K 继续发送。" 握手和挥手的过程，本质上就是双方交换和确认序列号的过程。

三次握手（Three-Way Handshake）

建立 TCP 连接的完整过程
①

第一次握手：客户端发送 SYN

客户端想要建立连接，向服务器发送一个 SYN（同步）报文段。 报文中携带客户端的初始序列号 seq = x（随机生成）。 此时客户端进入 SYN_SENT 状态，意思是"我发了同步请求，正在等回复"。

SYN = 1, seq = x
②

第二次握手：服务器回复 SYN+ACK

服务器收到 SYN 后，同意建立连接，于是回复一个 SYN+ACK 报文段。 其中 SYN = 1 表示服务器也想同步自己的序列号，ACK = 1 表示确认收到了客户端的请求。 服务器携带自己的初始序列号 seq = y，确认号 ack = x+1（意思是"我期待你下一个发 x+1"）。 服务器进入 SYN_RCVD 状态。

SYN = 1, ACK = 1, seq = y, ack = x+1
③

第三次握手：客户端发送 ACK

客户端收到服务器的 SYN+ACK 后，发送一个 ACK 报文段作为确认。 携带 seq = x+1（因为上一次发了 x，这次 +1），确认号 ack = y+1（告诉服务器"我收到了你的 y"）。 发送完毕后，客户端进入 ESTABLISHED 状态。服务器收到后也进入 ESTABLISHED 状态。 至此，连接正式建立！

ACK = 1, seq = x+1, ack = y+1

📞 生活类比：打电话

想象你（客户端）给朋友（服务器）打电话：

🧑
你（客户端）
"喂？你能听到我说话吗？" → SYN
👨‍💻
朋友（服务器）
"我能听到！那你能听到我说话吗？" → SYN+ACK
🧑
你（客户端）
"我也能听到你！那我们开始聊吧。" → ACK
为什么是"三次"而不是"两次"？

如果只有两次握手，服务器在收到第一个 SYN 后就进入 ESTABLISHED 状态，但此时它并不知道客户端是否真的收到了自己的回复。 如果客户端的 SYN 因为网络延迟重复到达，服务器会误以为客户端又发起了一次连接，从而浪费资源。 第三次握手确保了双方的发送和接收能力都正常，并且双方的初始序列号都被对方确认。

👋

四次挥手（Four-Way Handshake）

断开 TCP 连接的完整过程
①

第一次挥手：客户端发送 FIN

客户端数据发送完毕，想要关闭连接，发送一个 FIN（结束）报文段，携带序列号 seq = m。 此时客户端进入 FIN_WAIT_1 状态，意思是"我说完了，等你确认"。 注意：客户端发送 FIN 后，仍然可以接收服务器发来的数据。

FIN = 1, seq = m
②

第二次挥手：服务器回复 ACK

服务器收到 FIN 后，回复一个 ACK 报文，确认号 ack = m+1。 此时服务器进入 CLOSE_WAIT 状态，客户端收到后进入 FIN_WAIT_2 状态。 这一步只是确认收到了客户端的关闭请求，服务器可能还有数据没发完！所以 TCP 是"半关闭"的——客户端不再发送数据，但服务器还可以继续发。

ACK = 1, ack = m+1
③

第三次挥手：服务器发送 FIN

服务器数据也发送完毕了，发送自己的 FIN 报文段，携带序列号 seq = n。 服务器进入 LAST_ACK 状态，意思是"我也说完了，等你最后确认"。

FIN = 1, ACK = 1, seq = n, ack = m+1
④

第四次挥手：客户端回复 ACK

客户端收到服务器的 FIN 后，发送最后的 ACK 报文，确认号 ack = n+1。 客户端进入 TIME_WAIT 状态，等待 2MSL（最大报文段生存时间，通常 2 分钟）后才能真正关闭。 服务器收到 ACK 后直接进入 CLOSED 状态。

ACK = 1, seq = m+1, ack = n+1

📞 生活类比：挂电话

想象你和朋友通电话，聊完了要挂断：

🧑
你（客户端）
"我说完了，准备挂了。" → FIN
👨‍💻
朋友（服务器）
"好的，我知道了。等一下，我还有最后一句要说……" → ACK（半关闭）
👨‍💻
朋友（服务器）
"好了，我也说完了，可以挂了。" → FIN
🧑
你（客户端）
"好的，再见！" → ACK
（你等了一会儿确认对方真的挂了，才放下电话）
为什么挥手是"四次"而不是"三次"？

因为 TCP 是全双工通信——双方都可以同时发送和接收数据。 当客户端发送 FIN 时，只是表示"我不再发送数据了"，但服务器可能还有数据要发。 所以服务器需要先 ACK 确认，等自己的数据也发完后，再单独发一个 FIN。 这就是第二次和第三次挥手不能合并的原因——中间可能有一个数据传输的过程。

为什么要有 TIME_WAIT 状态？

客户端在发送最后一个 ACK 后，不直接关闭，而是等待 2MSL（约 2 分钟）。原因有两个：

1. 确保最后一个 ACK 到达服务器。如果 ACK 丢失，服务器会重发 FIN，客户端可以重新发送 ACK。

2. 让网络中残留的报文段过期。等待 2MSL 后，本次连接产生的所有报文都会从网络中消失，避免影响新连接。
